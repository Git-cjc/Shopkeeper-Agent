# 电商问数面试提纲

这份文档不是按教程章节整理，而是按面试中最容易被问到的问题整理。目标是帮助你在 3 到 5 分钟里，把项目讲清楚。

## 1. 这个项目到底解决什么问题

它解决的是“业务同学不会写 SQL，但又想直接通过自然语言查数”的问题。

核心难点不是把问题丢给模型生成一条 SQL，而是：

- 模型不知道企业里有哪些表和字段
- 模型不理解指标口径
- 模型可能不知道真实字段取值
- 模型生成的 SQL 可能语法错、字段错、口径错

所以这个项目的真正价值是：先把可用上下文召回出来，再让模型在更受约束的环境里生成 SQL。

## 2. 为什么不能直接让大模型猜 SQL

因为企业问数最大的问题不是“模型不会写 SQL 语法”，而是“模型不知道该基于什么业务上下文写 SQL”。

如果直接让模型猜，会有几个明显风险：

- 选错表
- 选错字段
- 误解指标含义
- 把真实值域猜错

所以这里先构建元数据知识库，再做召回和过滤，本质上是在给模型补“企业内部知识和上下文约束”。

## 3. 为什么要元数据知识库

元数据知识库是这套系统的基础设施，不是附属品。

它保存的不是最终查询结果，而是：

- 表信息
- 字段信息
- 指标信息
- 字段和指标的关系
- 字段示例值

有了它，系统才能先确定“可以用哪些业务对象”，再去做 SQL 生成，而不是把一切交给模型临场发挥。

## 4. 为什么要做混合检索

这个项目不是只有一种检索对象。

- 字段、指标适合做语义检索
- 字段真实取值适合做全文检索

所以这里拆成三路召回：

- 字段召回
- 指标召回
- 字段取值召回

这样做的原因是：

- 语义检索擅长理解“意思接近”
- 全文检索擅长命中“真实文本值”

两者结合，才能更贴近真实问数场景。

## 5. 为什么用 LangGraph，而不是一个大函数

因为这个系统天然就是一个多阶段流程：

- 先抽关键词
- 再做多路召回
- 再合并和过滤上下文
- 再生成 SQL
- 再校验、修正和执行

用 `LangGraph` 的好处是：

- 节点边界清晰
- 分支逻辑清晰
- 每一步都更容易调试
- 可以流式暴露进度

特别是 `validate_sql -> correct_sql -> run_sql` 这条闭环，用图结构表达会比手写大串 `if/else` 更稳定。

## 6. 为什么要有 QueryService 这一层

`QueryService` 的职责不是重写 Agent，而是做“API 和 Agent 之间的桥接”。

它主要做三件事：

- 创建本次请求的 `State`
- 组装运行所需的 `Context`
- 把图里产出的结构化消息包装成 SSE 文本

这层的价值是把：

- HTTP 协议
- LangGraph 运行
- SSE 输出

这三件事解耦开。

## 7. 为什么要用 SSE，而不是普通 JSON 接口

因为问数链路不是瞬间完成的，用户需要看到过程，而不只是最后结果。

这里的 SSE 统一返回三类消息：

- `progress`：当前执行到哪一步
- `result`：最终查询结果
- `error`：最终错误信息

这样前端就能实时展示：

- 哪个节点开始了
- 哪个节点成功了
- 哪个节点失败了
- 最后到底返回了什么

## 8. 为什么要 request_id

这是一个非常典型的后端工程问题。

多个请求并发时，日志会交错输出。如果没有请求级别的唯一标识，你很难快速排查某次问数到底在哪一步失败。

这里的做法是：

- `main.py` 中间件为每个请求生成 `request_id`
- `ContextVar` 保存当前请求上下文
- `loguru` 在每条日志输出前自动注入 `request_id`

这样就能按请求维度回放完整日志链路。

## 9. 这个系统最容易出错的地方在哪里

最容易出错的地方通常有三个：

### 1. 召回阶段

问题描述和元数据表述差异太大时，字段或指标可能召回不准。

### 2. SQL 生成阶段

即使召回信息基本正确，模型也可能把字段组合错、聚合写错，或者漏掉过滤条件。

### 3. SQL 校验和执行阶段

常见问题包括：

- 表名或字段名不正确
- SQL 方言细节有偏差
- 最终执行语句性能或结果规模不受控

## 10. 出错时怎么排查

我的排查顺序会是：

1. 看前端 SSE 消息停在了哪一步
2. 看对应步骤的 `progress` 和最终 `error`
3. 用 `request_id` 去后端日志里串完整链路
4. 判断问题出在：
   - 召回不足
   - 上下文构建有偏差
   - SQL 生成错误
   - SQL 校验/执行失败

这个顺序的好处是：先缩小范围，再看具体日志，而不是一上来就盲查所有节点。

## 11. 这个项目最像企业项目的地方是什么

我会强调三点：

- 不是模型直出 SQL，而是有元数据知识库和混合检索
- 不是一个 Prompt，而是完整的多节点工作流
- 不是只要跑通，还考虑了 SSE 交付和 request_id 日志追踪

这三点能把它和很多“套壳 Agent Demo”区分开。

## 12. 你后续还会怎么升级它

我会优先补两块：

### 1. SQL 安全防护

- 只读校验
- 危险关键词拦截
- 结果行数限制
- 超时控制

目标是让问数系统更像企业内部可控工具。

### 2. 问数评测集

- 先做 20 条，再扩到 50 条
- 统计 SQL 生成成功率、执行成功率、结果正确率
- 对召回质量做人工检查

目标是让系统不只是“能演示”，还可以被系统化评估。
