# 电商问数演示脚本

这份文档用于面试演示或项目讲解，目标是用固定问题快速覆盖系统的核心能力。建议优先演示 3 个问题，时间够再补第 4 个。

## 场景 1：简单聚合

### 用户问题

统计华北地区的销售总额。

### 预计触发的路径

- 抽取关键词：华北、销售总额
- 指标召回：销售总额
- 字段值召回：华北
- 表和字段过滤
- 生成聚合 SQL
- 校验并执行 SQL

### 适合展示的亮点

- 最基础的问数主链路是否打通
- SSE 进度是否会逐步返回
- 最终结果表格是否正确展示

### 可能被追问的技术点

- “销售总额”为什么是指标而不是普通字段
- “华北”是怎么被识别成真实字段值的
- 为什么最终不是让模型直接猜一条 SQL

## 场景 2：带维度过滤

### 用户问题

统计 2024 年黄金会员的订单总数。

### 预计触发的路径

- 抽取关键词：2024 年、黄金会员、订单总数
- 指标召回：订单总数
- 字段值召回：黄金会员
- 日期相关上下文补充
- 生成带时间过滤和会员等级过滤的 SQL
- 校验并执行 SQL

### 适合展示的亮点

- 系统是否能同时理解时间约束和业务维度值
- 字段值召回对过滤条件的帮助
- SQL 生成前的上下文补全作用

### 可能被追问的技术点

- 时间条件是在哪一层补进去的
- 如果“黄金会员”没有命中值域会怎样
- 日期过滤为什么不能完全交给模型自己猜

## 场景 3：带指标口径

### 用户问题

查询各地区客单价并按高到低排序。

### 预计触发的路径

- 抽取关键词：地区、客单价、排序
- 指标召回：客单价
- 表和字段过滤
- 生成带 `group by`、排序和指标计算逻辑的 SQL
- 校验并执行 SQL

### 适合展示的亮点

- 指标口径如何影响 SQL 生成
- 分组和排序类问题如何通过工作流落地
- 最终结果表格是否可直接用于业务分析

### 可能被追问的技术点

- 客单价是怎么定义的
- 指标和底层字段是什么关系
- 为什么指标信息要提前结构化维护

## 场景 4：更依赖字段值召回

### 用户问题

统计数码品类在上海地区的销售额。

### 预计触发的路径

- 抽取关键词：数码品类、上海、销售额
- 指标召回：销售额
- 字段值召回：数码品类、上海
- 过滤和补充候选字段
- 生成 SQL 并执行

### 适合展示的亮点

- 一个问题里同时包含多个真实值域
- 混合检索为什么比单纯向量召回更稳
- 字段值命中后如何影响最终过滤条件

### 可能被追问的技术点

- 值域信息为什么放在 Elasticsearch
- 多个值域命中时怎么避免上下文噪声过大
- 检索召回质量差时最先影响哪一步

## 场景 5：适合引出失败与纠错机制

### 用户问题

统计近 30 天高价值会员在重点地区的复购率。

### 预计触发的路径

- 抽取关键词：近 30 天、高价值会员、重点地区、复购率
- 指标召回：复购率
- 多个业务值召回
- 生成 SQL
- 可能在 `validate_sql` 阶段发现字段、口径或 SQL 结构问题
- 如有需要进入 `correct_sql`
- 再执行最终 SQL

### 适合展示的亮点

- 不是所有问题都一次生成成功
- 系统具备“校验 -> 修正 -> 再执行”的闭环
- 出错时前端和日志如何一起帮助排查

### 可能被追问的技术点

- 什么错误会进入 `correct_sql`
- 什么错误应该直接终止而不是修正
- 前端如何知道是步骤失败还是整次请求失败

## 演示顺序建议

如果只有 3 到 5 分钟，建议按下面顺序演示：

1. 场景 1：先证明主链路能跑通
2. 场景 2 或场景 4：展示字段值召回和上下文补全
3. 场景 5：展示异常与纠错闭环

如果有 10 分钟左右，可以再补：

4. 场景 3：展示指标口径和排序能力

## 演示时的讲解重点

演示时不要只盯结果表格，更要顺手讲清三件事：

- 这个问题触发了哪些召回和推理步骤
- 为什么这个系统比“直接让模型写 SQL”更稳
- 出错时你怎么通过 `SSE + request_id` 排查问题
