# 第十七章学习教程：前后端联调与日志追踪

这一章是前面 API 章节的收尾章。

如果前几章解决的是：

`接口怎么建起来，真实问数工作流怎么接进去。`

那么第17章解决的是：

`这个接口怎么真正给前端用，出错后怎么查，多个请求并发时怎么定位问题。`

如果只记一句话，请记这句：

`第17章的核心不是“多打几条日志”，而是给流式问数接口补齐交付能力和排障能力。`

---

## 1. 这一章在整条链路里的位置

先把本章主线压缩成一条线：

```text
前端页面 / Apifox
  -> POST /api/query
  -> QueryService.query(...)
  -> LangGraph 节点持续产出进度消息
  -> 后端统一包装成 SSE 文本
  -> 前端实时展示 progress / result / error
  -> 日志用 request_id 串起整次请求
```

这一章你真正要补上的，不是“能跑”而是“好用”：

- 前端要知道当前执行到哪一步
- 出错时前端要拿到失败原因
- 后端要知道日志属于哪一次请求

---

## 2. 这一章到底在解决什么问题

真实联调时，最容易遇到三个问题：

1. 某个节点报错了，后面的节点还要不要继续跑
2. 前端怎么区分“执行中”“成功”“失败”“最终结果”
3. 多个请求同时进来时，日志怎么知道谁是谁

所以这一章其实是在给第16章补工程闭环：

- 用统一 SSE 协议跟前端说话
- 用统一异常处理让错误能被看见
- 用 `request_id` 让日志可追踪

## 4. 先抓住这章最核心的协议：SSE 三类消息

这一章最重要的基础，不是某个函数，而是后端和前端之间约定好的消息格式。

项目里约定三类消息：

- `progress`：步骤进度
- `result`：最终结果
- `error`：最终错误

你可以把它理解成：

```text
progress = 系统现在做到哪了
result   = 最终查出了什么
error    = 最终为什么失败
```

### 4.1 `progress` 负责告诉前端“过程”

例如：

```json
{
  "type": "progress",
  "step": "抽取关键词",
  "status": "running"
}
```

这里的 `status` 一般有三种：

- `running`：执行中
- `success`：成功
- `error`：失败

前端看到它，就能把步骤状态渲染出来。

### 4.2 `result` 负责告诉前端“结果”

例如：

```json
{
  "type": "result",
  "data": [{ "region_name": "华北", "amount": 41099.5 }]
}
```

这个 `data` 一般就是数据库返回的结果列表。

### 4.3 `error` 负责告诉前端“失败原因”

例如：

```json
{
  "type": "error",
  "message": "division by zero"
}
```

要特别分清这两个错误概念：

- `progress.status = error`：哪一步失败了
- `type = error`：具体为什么失败

前者偏页面状态，后者偏异常详情。

## 5. `QueryService` 是前后端联调的核心桥梁

文件：

- [app/services/query_service.py](file:///Users/cjc/Documents/Code/AI-Agent-Learning/shopkeeper-agent/app/services/query_service.py)

这一章最该反复看的文件，就是它。

因为它在做一件很关键的事：

`把 LangGraph 内部流，翻译成前端能消费的 SSE 文本流。`

主线很简单：

```text
接收 query
  -> 创建 DataAgentState
  -> 创建 DataAgentContext
  -> 调用 graph.astream(...)
  -> 拿到 chunk 字典
  -> 转成 JSON 字符串
  -> 包成 data: ...\n\n
  -> yield 给 StreamingResponse
```

你要掌握这句话：

`QueryService 不是业务节点本身，而是 API 层和 Agent 层之间的翻译层。`

### 5.1 为什么这里必须做 JSON 序列化

节点里 `writer(...)` 写出的其实是 Python 字典，不是浏览器直接能吃的文本。

所以 `QueryService` 要负责：

```python
json.dumps(chunk, ensure_ascii=False, default=str)
```

这里两个参数很关键：

- `ensure_ascii=False`
  作用：保留中文，不把“抽取关键词”变成 Unicode 转义
- `default=str`
  作用：如果结果里有日期、Decimal 这类 JSON 不认识的类型，先转成字符串兜底

### 5.2 为什么异常不能只打印日志

这一章一个非常重要的工程意识是：

`流式接口出了错，不能只在后端自己知道。`

因为前端还在等消息。

所以 `QueryService` 的异常兜底逻辑是：

```text
节点 raise
  -> graph.astream(...) 中断
  -> QueryService 捕获异常
  -> 产出 {"type": "error", "message": "..."}
  -> 继续通过 SSE 发给前端
```

这就是为什么它不是简单返回 HTTP 500。

原因很现实：

`流式响应一旦开始写出，HTTP 状态码通常已经发出去了，后面很难再改。`

所以最稳的办法，就是：

`把错误也当成一条 SSE 消息发出去。`

## 6. 节点异常处理要分两种情况理解

这一章特别容易学混的点是：

`为什么有的异常要继续 raise，有的异常却只是写回 state。`

你要分成两类：

### 6.1 系统异常：要中断整条链路

比如：

- 召回时报错
- SQL 执行时报错
- 程序逻辑本身报错

这种情况要走：

```text
writer progress error
  -> logger.error(...)
  -> raise
  -> graph 停止
  -> QueryService 输出 type=error
```

也就是说：

`节点告诉前端“这一步失败了”，同时继续抛异常，让整条错误链路停下来。`

### 6.2 业务分支异常：不一定要中断

典型例子就是 `validate_sql`。

它执行 `EXPLAIN` 时失败，很多时候不是系统崩了，而是：

`SQL 需要进入下一步校正。`

这时更合理的做法不是直接 `raise`，而是：

```text
把错误信息写进 state["error"]
  -> 交给条件边判断
  -> 进入 correct_sql 节点
```

所以你要记住一句判断标准：

`系统异常要中断，业务分支要流转。`

## 8. `request_id` 是这一章日志部分的主角

当只有一个请求时，日志很好看。

但真实场景里日志会交错：

```text
请求A：抽取关键词成功
请求B：抽取关键词成功
请求A：召回字段成功
请求C：生成SQL成功
```

这时候如果没有请求标识，你很难把一整次调用链路重新拼出来。

所以这一章做了三步：

```text
请求进入
  -> 中间件生成 request_id
  -> 写进 ContextVar
  -> loguru 每次打日志时自动读出来
  -> 每条日志都带同一个 request_id
```

这个设计你一定要吃透，因为它在 Web 项目里非常常见。

## 9. `main.py`：用中间件给每次请求打编号

文件：

- [main.py](file:///Users/cjc/Documents/Code/AI-Agent-Learning/shopkeeper-agent/main.py)

第17章在这里做的事很简单，但价值很大：

```python
@app.middleware("http")
async def add_request_id(request: Request, call_next):
    request_id = uuid.uuid4()
    request_id_ctx_var.set(request_id)
    response = await call_next(request)
    return response
```

这里你要知道几个词：

- `middleware`：中间件
  含义：在真正进入路由前后插一层统一处理逻辑
- `uuid`
  含义：通用唯一标识
  这里用来给每次请求生成唯一编号
- `call_next`
  含义：把请求继续往后交给真正的路由处理函数

这一层的本质就是：

`所有请求先统一领一个号，再继续处理。`

## 10. `ContextVar`：为什么不用普通全局变量

文件：

- [app/core/context.py](file:///Users/cjc/Documents/Code/AI-Agent-Learning/shopkeeper-agent/app/core/context.py)

这是第17章另一个必须掌握的知识点。

很多初学者第一反应会是：

`我定义一个全局变量 current_request_id 不就行了？`

单请求时看起来可以，但并发时会出问题。

原因很简单：

```text
请求A 把全局变量改成 A
请求B 又把全局变量改成 B
请求A 后面再读时，拿到的可能已经是 B
```

所以普通全局变量不适合异步 Web 请求上下文。

### 10.1 `ContextVar` 可以怎么理解

你可以把它理解成：

`带隔离能力的全局变量。`

更准确一点说：

- `ContextVar`：上下文变量
- 在 FastAPI 这种异步程序里，可以理解成“每个请求各自有一份值”

如果你学过 Java，也可以把它类比成：

`ThreadLocal 在异步场景下的近亲。`

### 10.2 默认值为什么重要

项目里这里写了：

```python
request_id_ctx_var: ContextVar[str] = ContextVar("request_id", default="1")
```

这个默认值不是随便写的。

因为不只是 HTTP 请求会打日志，脚本启动、初始化流程也可能打日志。

这些场景不经过中间件，没有人为它提前设置 `request_id`。

所以要给一个默认值，避免日志读取时报错。

## 11. `log.py`：日志器为什么能自动带上 request_id

文件：

- [app/core/log.py](file:///Users/cjc/Documents/Code/AI-Agent-Learning/shopkeeper-agent/app/core/log.py)

这一章日志设计最值得你掌握的点是：

`业务代码根本不用手动拼 request_id。`

它能自动带上，是因为日志模块统一做了注入。

核心思路：

```text
logger.info(...)
  -> loguru 输出前先经过 patch
  -> patch 函数读取 ContextVar
  -> 写入 record["extra"]["request_id"]
  -> format 用 {extra[request_id]} 展示出来
```

这里几个名词要理解：

- `record`
  含义：一条日志在输出前的结构化记录
- `extra`
  含义：日志框架预留的附加字段区域
- `patch`
  含义：给日志器加一层统一的增强逻辑

所以 `inject_request_id(record)` 其实就是：

`在每条日志正式输出前，把当前请求编号塞进去。`

## 14. 学这一章时最容易踩的坑

- 以为 `progress.status=error` 就等于最终错误返回
  实际不是。它只表示某一步失败了，最终错误详情还要看 `type=error`。
- 以为节点里 `except` 住异常就算处理好了
  实际很多情况下还要继续 `raise`，不然图会错误地往后执行。
- 以为日志注入是每次 `logger.info(..., request_id=xx)` 手动传的
  实际这里是统一通过 `ContextVar + patch` 自动注入。
- 以为 `ContextVar` 只是“高级全局变量”
  实际它最关键的价值是并发隔离。

## 15. 这一章学习建议

我的建议不是背代码，而是做三次“顺链路复述”：

### 第一次：只复述前后端消息链路

试着不看代码，把这条线说出来：

```text
前端发请求
  -> QueryService 调 graph.astream
  -> 节点 writer 写出 progress/result
  -> QueryService 包成 SSE 文本
  -> 前端按 type 渲染
```

### 第二次：只复述异常链路

试着自己说：

```text
节点失败
  -> writer progress error
  -> raise
  -> QueryService 捕获
  -> 返回 type=error
```

### 第三次：只复述日志链路

试着自己说：

```text
请求进入 main.py
  -> 中间件生成 request_id
  -> ContextVar 保存
  -> loguru patch 读取
  -> 每条日志自动带 request_id
```

如果你这三条都能不看代码讲顺，这一章就真的吃下来了。
