# 第十六章学习教程：查询接口实现与依赖组装

这一章是第十五章的自然延续。

第十五章你学到的是：

`FastAPI 的 API 壳子怎么搭起来，SSE 流式响应怎么跑通。`

第十六章要做的是：

`把第15章里的假流式接口，换成真实的问数工作流，并把依赖装配这件事做规范。`

如果只记一句话，请记这句：

`第16章的核心不是“又多写了几个 Depends 函数”，而是把 HTTP 请求正式接进 LangGraph 问数智能体。`

## 1. 这一章在整条链路里的位置

先把当前对外调用链路压缩成一条线：

```text
前端 / Apifox
  -> POST /api/query
  -> query_router.py 接收请求
  -> Depends(get_query_service) 获取 QueryService
  -> QueryService.query(...)
  -> graph.astream(...)
  -> StreamingResponse 按 SSE 格式持续返回
```

和第15章对比，你要抓住一个核心变化：

- 第15章：先把流式 API 外壳跑通
- 第16章：把真实问数工作流接进流式 API

也就是说，第16章开始，接口不再只是“会流式返回”，而是：

`真的能驱动问数智能体。`

## 4. 先从路由看：请求是怎么进来的

文件：

- [app/api/routers/query_router.py](file:///Users/cjc/Documents/Code/AI-Agent-Learning/shopkeeper-agent/app/api/routers/query_router.py)

这一章的路由层做的事情很少，但分工非常清晰：

```text

2. 通过 Depends 拿到 QueryService
3. 把 QueryService.query(...) 交给 StreamingResponse
```

最该掌握的不是 FastAPI 语法，而是边界：

`路由层只处理 HTTP 协议相关的事，不直接 new Repository，也不直接调图节点。`

## 5. `QueryService` 才是这一章的业务核心

文件：

- [app/services/query_service.py](file:///Users/cjc/Documents/Code/AI-Agent-Learning/shopkeeper-agent/app/services/query_service.py)

这一章最核心的变化，其实就是引入了真正的 `QueryService`。

它负责把一次 HTTP 请求，转换成一次 LangGraph 问数执行。

它的主线非常清楚：

```text
接收 query 字符串
  -> 创建 DataAgentState
  -> 创建 DataAgentContext
  -> 调用 graph.astream(...)
  -> 把内部 chunk 包成 SSE 消息
  -> 持续 yield 给路由层
```

这里你要特别注意一句话：

`QueryService 不是在重写 Agent，而是在做 API 层和 Agent 层之间的桥接。`

它之所以必要，是因为路由层不适合直接处理这些细节：

- State 怎么创建
- Context 怎么组装
- graph.astream 怎么消费
- SSE 文本怎么封装
- 异常怎么变成流式错误消息

## 6. `state` 和 `context` 在 API 层是怎么落地的

这一章有一个很好的复习机会：把你前面学过的 `DataAgentState` 和 `DataAgentContext` 放回真实调用链里看。

### 6.1 `state` 是这次问数任务会变化的数据

例如：

- `query`
- `keywords`
- `retrieved_column_infos`
- `table_infos`
- `sql`
- `error`

这些都会随着图执行不断更新。

### 6.2 `context` 是节点执行时要用的工具

例如：

- `embedding_client`
- `column_qdrant_repository`
- `metric_qdrant_repository`
- `value_es_repository`
- `meta_mysql_repository`
- `dw_mysql_repository`

这些对象不是业务结果，而是：

`本次图执行时要拿来用的外部工具。`

所以第16章第一次把你前面学过的 state/context 设计，真正放进 API 查询链路里了。

## 8. 依赖树怎么理解

这一章可以把最终依赖树记成这样：

```text
get_query_service
  -> get_meta_mysql_repository
      -> get_meta_session
  -> get_embedding_client
  -> get_dw_mysql_repository
      -> get_dw_session
  -> get_column_qdrant_repository
  -> get_metric_qdrant_repository
  -> get_value_es_repository
```

你可以从中看出两层资源生命周期：

### 8.1 应用级资源

这些对象在服务启动时初始化一次：

- Qdrant client
- Embedding client
- ES client
- MySQL client manager

### 8.2 请求级资源

这些对象是每次请求里创建和释放的：

- MySQL Session
- 基于 Session 的 Repository
- 基于多个依赖组装出的 QueryService

这就是为什么网站原文会同时强调：

- `lifespan`
- `Depends`

因为它们分别管不同层级的问题。

## 9. `lifespan.py` 在解决什么问题

文件：

- [app/api/lifespan.py](file:///Users/cjc/Documents/Code/AI-Agent-Learning/shopkeeper-agent/app/api/lifespan.py)

这一章里 `lifespan` 的价值非常实际：

`外部客户端不应该每个请求都重新初始化。`

因为这些资源通常比较重：

- 要建网络连接
- 可能有连接池
- 初始化成本高

所以更合理的方式是：

```text
应用启动时初始化
  -> 请求期间复用
  -> 应用关闭前统一释放
```

网站原文其实就是在帮你建立一个工程意识：

`应用级资源和请求级资源要分层管理。`

---

## 10. 这一章里的 SSE 到底发生在哪一层

很多人容易把“流式接口”误以为完全是路由层的事。

其实第16章里，SSE 是分两层完成的：

### 10.1 QueryService 负责把内部 LangGraph 流转成 SSE 文本

例如：

```python
yield f"data: {json.dumps(chunk, ensure_ascii=False, default=str)}\n\n"
```

这里完成的是：

`内部 chunk -> SSE 格式字符串`

### 10.2 路由层负责把这个生成器交给 StreamingResponse

例如：

```python
return StreamingResponse(
    query_service.query(query.query),
    media_type="text/event-stream",
)
```

这里完成的是：

`SSE 生成器 -> HTTP 流式响应`

你一定要分清这两个层次。

## 11. 为什么异常也要包装成 SSE

普通 JSON 接口里，异常通常可以直接：

- 抛出
- 返回 500

但流式接口不一样。

一旦已经开始往外返回数据，HTTP 状态码通常就已经发出去了。  
这时如果中途报错，就不能再改成一个标准 JSON 错误响应。

所以这里的做法是：

`把异常也包装成一条 SSE 消息返回给前端。`

这也是第16章非常值得掌握的点：

`流式接口的错误处理方式，和普通接口不同。`

## 14. 这一章最容易学偏的地方

### 14.1 把 `dependencies.py` 当成业务代码

它不是在写“问数逻辑”，而是在写“怎么把对象装起来”。

### 14.2 以为 `QueryService` 只是多包了一层

不是。

它是 API 层和 Agent 层之间的桥梁，负责：

- state/context 组装
- 流式输出转换
- 异常包装

### 14.3 分不清应用级资源和请求级资源

这是第16章最典型的难点。

- `lifespan` 管应用级资源
- `Depends` 常用于请求级对象装配

如果这个边界不清楚，后面代码会越来越乱。
