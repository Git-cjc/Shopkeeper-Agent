# 电商问数项目总览

这个项目面向的不是“让大模型直接猜 SQL”，而是一个更接近企业真实问数场景的后端系统：

- 用户输入自然语言问题
- 系统先召回相关字段、指标和字段取值
- 再基于上下文生成、校验、修正并执行 SQL
- 最后以流式方式把进度、结果和错误返回前端

这个仓库基于公开教程体系完成复现与理解沉淀，并在此基础上继续朝“可讲、可扩展、可工程化”的方向整理。

## 系统分层

### 1. 数据层

- 教学数仓 `MySQL`：提供事实表、维度表和最终 SQL 执行环境
- 元数据库 `MySQL`：保存表、字段、指标、字段指标关系等结构化元数据

这一层解决的是“真实数据在哪里”和“权威结构信息在哪里”这两个问题。

### 2. 检索层

- `Qdrant`：做字段和指标的语义召回
- `Elasticsearch`：做字段真实取值的全文检索

这一层解决的是“不要让模型凭空猜字段、猜指标、猜业务取值”。

### 3. Agent 层

- `LangGraph`：组织关键词抽取、多路召回、信息合并、过滤补全、SQL 生成、校验、修正、执行
- 各节点职责单一，便于调试和扩展

这一层解决的是“如何把一次问数任务拆成稳定、可观察、可纠错的步骤”。

### 4. 服务层

- `QueryService`：负责把 API 请求转换成一次 LangGraph 执行
- 组装 `DataAgentState` 和 `DataAgentContext`
- 把内部结构化消息统一包装成 SSE 文本流

这一层解决的是“如何把 Agent 能力接进真实接口”。

### 5. API 层

- `FastAPI`：提供 `/api/query`
- `Depends` 负责依赖注入
- `lifespan` 负责生命周期资源管理

这一层解决的是“如何把系统能力稳定暴露给前端或外部调用方”。

### 6. 前端层

- `React + Vite + Tailwind CSS`
- 负责输入问题、展示步骤进度、展示最终查询结果和错误

这一层解决的是“用户怎么用，以及过程怎么被看见”。

### 7. 可观测层

- `SSE`：实时返回 `progress / result / error`
- `ContextVar + loguru`：为并发请求注入 `request_id`

这一层解决的是“联调时能不能看到过程，出错时能不能快速定位”。

## 调用链路

这套系统的主链路可以压缩成下面这条线：

```text
用户问题
  -> 关键词抽取
  -> 字段 / 指标 / 字段取值多路召回
  -> 召回结果合并
  -> 表和指标过滤
  -> 补充数据库上下文
  -> 生成 SQL
  -> EXPLAIN 校验 SQL
  -> 必要时修正 SQL
  -> 执行 SQL
  -> QueryService 统一包装成 SSE
  -> 前端展示 progress / result / error
```

这里最关键的工程思想不是“调一次模型”，而是：

- 先检索，再生成
- 先校验，再执行
- 先拆步骤，再做流式交付

## 技术选型原因

### 为什么要 MySQL 元数据库

元数据知识库不是给用户查结果的，而是给系统理解“有哪些表、字段、指标、口径关系”。

如果没有这一层，模型只能靠提示词和自身记忆去猜，容易出现：

- 选错表
- 选错字段
- 误解指标口径
- SQL 幻觉

把结构化元数据单独落到 `MySQL`，可以让系统先拿到确定性的业务骨架，再去做后续推理。

### 为什么要 Qdrant + Elasticsearch 混合检索

问数场景里的“相关信息”不止一种：

- 字段、指标更适合做语义检索
- 真实字段值更适合做关键词或全文匹配

如果只做向量检索，像“华北”“黄金会员”这类真实取值可能不稳。
如果只做全文检索，语义相近但表述不同的问题又容易召回不到。

所以这里拆成：

- `Qdrant` 负责语义理解
- `Elasticsearch` 负责值域命中

这是比单一路径更贴近企业问数实际的一种做法。

### 为什么要用 LangGraph

这个项目不是单轮 Prompt 调用，而是一条有明显步骤边界的业务链路：

- 某些步骤需要显式中断
- 某些步骤需要条件分支
- 某些步骤需要流式暴露执行进度

`LangGraph` 适合把这样的流程表达成图结构，而不是把全部逻辑揉进一个大函数里。

它的好处是：

- 节点职责清晰
- 失败位置清晰
- 条件边可控
- 更适合后续扩展纠错、评测和安全能力

### 为什么要用 SSE

问数链路不是瞬时完成的，前端需要知道系统现在做到哪一步。

`SSE` 的价值是：

- 可以持续返回节点进度
- 可以在同一条流里返回结果或异常
- 对前端展示步骤状态很友好

这比“等全部跑完一次性返回 JSON”更适合 Agent 类产品。

### 为什么要 request_id

一旦多个请求并发执行，日志会交错在一起。

如果没有 `request_id`，你很难快速判断某条日志属于哪次问数请求。通过：

- 中间件生成 `request_id`
- `ContextVar` 保存请求上下文
- `loguru` 自动注入日志字段

就能把同一次请求的整条链路重新串起来。

## 已完成的工程增强

### 1. SQL 安全防护

当前已经补齐以下能力：

- 只读校验：限制只允许 `SELECT` / `WITH ... SELECT`
- 危险关键词拦截：拦截 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`
- 结果行数限制：自动补齐或规范化 `LIMIT`
- 超时控制：避免慢 SQL 长时间占用资源
- 错误输出与日志记录：保持前端协议统一，同时能记录被拦截原因

这部分的价值在于：把“能跑通的问数系统”向“更像企业内部可控工具”的方向推进。

### 2. 问数评测集

当前已经建立一套可重复执行的离线评测样本，首版为 20 条中文问题，后续仍可扩展到 50 条。

重点评估：

- SQL 正确率（值正确）
- 严格结果一致率
- 列别名一致率
- SQL 执行成功率
- 表 / 字段 / 指标 / 取值召回覆盖率

当前这套 20 题评测样本的最新结果为：

- SQL 正确率（值正确）：`100%`
- 执行成功率：`100%`
- 表召回覆盖率：`100%`
- 字段召回覆盖率：`75%`
- 指标召回覆盖率：`100%`
- 取值召回覆盖率：`100%`

这部分的价值在于：证明系统不仅能演示，还能被系统化评估。
